# ネットワークスペシャリスト試験 試験プロフィール

この文書は、ネットワークスペシャリスト試験の出題範囲、問題形式、復習単元の切り方を整理し、`03_prompt_add_explanationText.md` と `04_prompt_link_questionSetId.md` の共通前提にするための資料である。

## 試験全体の見取り図
- ネットワークスペシャリスト試験は、IPA の高度試験区分に属するネットワーク設計・構築・運用・評価・改善提案の実務判断を問う試験である。
- `output/nw/questions_json` にある取得済みデータは、2009年春から2025年春までの午前I・午前IIの55問構成で、16回分・合計880問である。
- 問題形式は全問 `true_false` で、設問意図は `select_correct` が867問、`select_incorrect` が13問である。
- したがって、学習者に有益な解説は、単なる用語説明よりも、**どの観点で正誤判定するか**、**似た技術や運用概念とどう切り分けるか**を短く示す形になる。

## 午前Iと午前IIの性格
### 午前I
- 応用情報技術者試験相当の広い共通知識から出題される。
- 基礎理論、アルゴリズム、コンピュータ構成、データベース、セキュリティ、マネジメント、ストラテジなどが混在する。
- ただし、ネットワークスペシャリストの学習者にとっては、単独知識として終わらせず、**ネットワーク設計・運用の判断にどうつながるか**まで落とし込んで復習する方が再利用性が高い。

### 午前II
- ネットワーク、セキュリティ、性能、可用性、運用管理、クラウド、無線、標準化など、よりネットワーク実務に近い知識が中心になる。
- 単語暗記型に見えても、実際には**要件定義・設計・監視・障害対応・改善提案のどの場面の知識か**を見抜くことが重要である。

## シラバス上の主な大項目
`output/nw/category/category.json` では、IPA シラバス Ver.4.1 を基に次の7フォルダを採用している。

1. ネットワークシステムの要件定義
2. ネットワークシステムの設計
3. ネットワークシステムの構築とテスト
4. ネットワークシステムの運用・保守
5. ネットワークシステムの管理
6. ネットワークシステムの評価
7. 個別業務システム開発のコンサルティング

## この試験で頻出の判断軸
- その技術や設定が、どの要件を満たすためのものか
  - 可用性、性能、拡張性、運用性、セキュリティ、経済性
- 似た技術の役割分担を区別できているか
  - 例: `IPsec` と `TLS`、`ルータ` と `L3スイッチ`、`IDS` と `IPS`
- 設計論点と運用論点を混同していないか
  - 例: 冗長化方式の選択は設計、閾値監視や障害切り分けは運用管理
- 現象と対策の対応が正しいか
  - 例: 遅延、輻輳、再送、単一点障害、認証失敗、脆弱性対応
- 単なる知識問題に見えても、実務でどの場面に使う知識かを説明できるか

## 学習者がつまずきやすいポイント
- 規格名や略語を覚えていても、何を守る技術か、どの層で働くかが曖昧なまま解いてしまいやすい。
- 暗号化、認証、完全性、可用性のどれを達成する技術かを取り違えやすい。
- 設計時の要件定義と、障害発生後の運用対応を混同しやすい。
- 性能問題で、スループット、帯域、遅延、応答時間、待ち行列の指標を混同しやすい。
- 共通知識問題を単発暗記で処理し、ネットワーク実務との接点を作らないまま終わらせやすい。

## 解説文作成への示唆
- `explanationText` では、用語の辞書的定義だけで終わらせず、**何を実現する技術か**、**なぜ他の選択肢ではだめか**を書く。
- 正しい選択肢では、要件との対応関係を明示する。
- 誤りの選択肢では、誤っている語句、層、役割、適用場面を特定し、正しい内容へ置き換える。
- 午前Iの共通知識問題でも、ネットワークスペシャリスト受験者が復習単元へ接続できるよう、最後に一段ネットワーク文脈へ寄せると学習効果が高い。

## questionSetId 紐付けへの示唆
- `category` の原分類は IPA の広い知識区分だが、`questionSetId` は NW シラバス上の復習単元へ寄せて付与する。
- したがって、元の `categorySmall` をそのまま単純移送するのではなく、**受験者がどのネットワーク実務単元を復習すべきか**で判断する。
- ネットワーク方式、通信プロトコル、アドレス設計、無線、リモートアクセス、可用性設計の大半は `2-2 ネットワークシステムの設計` を主軸に考える。
- 監視、障害解析、性能分析、セキュリティ侵害対応は、設計論点と混同せず `5-x` へ寄せる。
