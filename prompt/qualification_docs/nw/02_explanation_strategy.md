# ネットワークスペシャリスト試験 解説文戦略

この文書は、`03_prompt_add_explanationText.md` でネットワークスペシャリスト試験の `explanationText` を書く際に、何を補うべきかを整理した補助資料である。

## 解説の狙い
- ネットワークスペシャリストの解説は、単なる略語や規格の説明で終わってはいけない。
- 常に次の3点を同時に狙う。
  1. この問題の正誤理由の理解
  2. ネットワーク設計・運用での使いどころの理解
  3. 類題で再利用できる見分け方の定着
- 良い解説は、「何が正しいか」だけでなく、「何を比較すれば誤答を落とせるか」まで示す。

## 参照の基本方針
- NWでは `20_merged_1` の `explanation_choice_snippets` を主素材にする。
- 取得済みデータでは `explanation_common_summary` は基本的に空で、選択肢ごとの断片説明が主な根拠になる。
- 元サイトの断片をそのまま貼り戻さず、判定軸をそろえて学習者向けに再構成する。
- 外部Webの再取得は禁止されている工程では、`question_url` に戻らずローカル JSON と資格別資料だけで完結させる。

## この試験の解説で最優先にすべきこと
- まず `正しい。` または `間違い。` を明示する。
- 次に、どの観点で判断したかを一文で示す。
  - 要件との対応
  - 役割の違い
  - 層の違い
  - 設定対象の違い
  - 運用フェーズの違い
- その後で、誤答なら「どこが違うか」「正しくは何か」を短く補う。

## NWで頻出のひっかけ
- 似た技術の役割を入れ替えている
  - 例: `VPN` と `TLS`、`IDS` と `IPS`、`RAID` とバックアップ
- 設計時に決める内容と、運用時に監視・対応する内容を混同している
- 可用性・性能・セキュリティ・運用性のどの要件を満たす技術かがずれている
- OSI参照モデルやTCP/IPの階層をまたいで誤った説明をしている
- 規格名は正しくても、適用場面や制約条件が誤っている
- 数式や待ち行列の問題で、指標の意味を取り違えている

## NWで「ハッとする」解説の条件
- その誤答がなぜもっともらしく見えるかを短く言語化している。
- 正しい技術を、対立候補と比較して説明している。
- 実務のどの場面で使う知識かが分かる。
- 略語暗記ではなく、要件と手段の対応で覚え直せる。

## 書き分けの原則
### 1. ネットワーク設計問題
- どの要件を満たす設計かを先に示す。
- 冗長化、アドレス設計、経路制御、負荷分散、無線設計では、目的を抜くと解説が弱くなる。
- 例:
  - `可用性を高めるための設計`
  - `ブロードキャストドメインを分割するための設計`
  - `暗号化ではなく認証強化が主目的の方式`

### 2. セキュリティ問題
- `機密性・完全性・可用性・真正性・否認防止` のどれに効くのかを明示する。
- 技術名だけでなく、`何をどこで守るか` を書く。
- 誤答では、防御対象、配置場所、検知か遮断かの違いを示す。

### 3. 性能・待ち行列・評価問題
- 何の指標を計算・比較しているかを先に言う。
- `帯域`、`スループット`、`遅延`、`応答時間`、`利用率` を混同しない。
- 計算過程を長く展開するより、判定に必要な式や関係だけを短く書く。

### 4. 運用・監視・障害対応問題
- 監視か、切り分けか、復旧か、再発防止かを明示する。
- 操作手順の丸暗記ではなく、どの順番で確認すべきかを書く。
- `原因特定前に設定変更へ走っていないか` を誤答判定の軸にしやすい。

### 5. 午前Iの共通知識問題
- 元分類がネットワーク以外でも、NW受験者にとっての意味付けを一段加える。
- 例:
  - 待ち行列理論は `5-3 システム性能の分析`
  - 情報セキュリティ一般は `2-2` か `5-4`
  - 標準化や新技術動向は `2-1` か `6-2`

## NWで避けるべき弱い解説
- `DNSの説明として正しい。`
- `セキュリティ上重要である。`
- `性能を改善できる。`
- `監視で使用する。`
- このような一般論だけでは、他の選択肢との差が見えない。

## NWで強い解説の型
- 1文目:
  - `正しい。` または `間違い。`
- 2文目:
  - 判定の直接理由
- 3文目:
  - 対立概念との違い、または適用場面
- 4文目が必要なら:
  - 類題での見分け方

例:
```json
"間違い。\n\n誤っているのは、TLS と IPsec の適用層と保護範囲を混同している点である。TLS は主にトランスポート層より上で通信セッションを保護し、IPsec はIP層でパケット単位に保護する。暗号化技術は、どの層でどの通信を守るかまで対応づけて覚える必要がある。"
```

## suggestedQuestions の方向性
- 受験者が次に押したくなる質問は、抽象的な `詳しく教えて` ではなく、比較軸があるものにする。
- 良い例:
  - `IPsec と TLS は何が違うのか`
  - `この方式が可用性向上に効く理由は何か`
  - `設計の論点と運用の論点はどう見分けるか`
  - `この問題で最初に見るべきキーワードは何か`

## lawGroundedExplanationNotNeeded の扱い
- NWでは法令条文からの追加解説が主論点になる問題は基本的に想定しにくい。
- そのため、通常は `lawGroundedExplanationNotNeeded: true` を基本に考えてよい。
- ただし、情報セキュリティ管理、個人情報保護、知的財産、標準化、契約・監査などの制度的背景を問う問題では、断定に迷う場合は `false` を残す。
